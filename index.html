História

História - Web services

  • Inicialmente, todo o processamento era centralizado
  • Evolução das redes de computadores
  • Aplicações distribuídas
  • Processamento começou a ser distríbuido
  • Objetivo: Integrar sistemas heterogênios

História - SOAP

  • Protocolo projetado para invocar aplicações remotas através do RPC...
  • Ou troca de mensagens, em um ambiente independente de plataforma e/ou linguagem de programação
  • Beneficia-se da infraestrutura do HTTP, o que facilita a comunicação entre Webservices
  • Sem características proprietárias
  • Passou a ser utilizado no lugar do CORBA e DCOM

Arquitetura

Arquitetura

Mensagem SOAP é constituída por:

  • Envelope
  • Header
  • Body

Arquitetura

Arquitetura - Envelope

Elemento principal do XML que representa a mensagem

Arquitetura - Header

Mecanismo genérico que permite a adição de características à mensagem SOAP

Arquitetura - Body

Contém a informação a ser transportada

Arquitetura - Exemplo

Arquitetura - Protocolo

  • SOAP Envelope
  • SOAP Encoding Rules
  • SOAP RPC Style
  • LINK SOAP HTTP

Arquitetura - SOAP Envelope

Define o que há na mensagem e como processá-la. Única parte obrigatória.

Arquitetura - SOAP Encoding Rules

Mecanismo de serialização, pode ser usado para troca de instâncias de tipos definidos pela aplicação.

Arquitetura - SOAP RPC Style

Define uma convenção que pode ser usada para representar as chamadas e respostas remotas aos procedimentos.

Arquitetura - LINK SOAP HTTP

Protocolo que liga o SOAP e o HTTP

Arquitetura - Atributos SOAP

  • Atributo encodingStyle
  • Atributo mustUnderstand
  • Atributo actor

Arquitetura - encodingStyle

Indica as regras de serialização usadas nas mensagens SOAP

Arquitetura - mustUnderstand

Indica se a entrada é obrigatória ou não

Arquitetura - actor

Usado pelo header para definir o último ponto de recepção de uma mensagem

Características

Características

  • Definido pelo consórcio W3C
  • Protocolo baseado em XML (sistemas distribuídos)
  • Padrão de utilização para Web Services
  • Normalmente utiliza HTTP como protocolo de transporte

Dependências e Compatibilidades

Dependências e Compatibilidades

O protocolo SOAP não está vinculado a nenhuma plataforma de hardware, software ou linguagem de programação, o que o torna compatível com diferentes sistemas operacionais e diferentes tecnologias.

Vantagens e desvantagens

Vantagens

  • Interoperabilidade entre sistemas
  • Permite a comunicação entre sistemas protegidos por firewalls
  • SOAP descreve completamente cada elemento na mensagem, facilitando o entendimento e a proteção contra erros.
  • Os conceitos sobre segurança são melhores especificados e difundidos nos protocolos baseados em SOAP do que no protocolo HTTP, utilizado pelo estilo REST.

Desvantagens

  • Limita-se ao método POST
  • É muito trabalhoso criar uma interface WSDL manualmente
  • Todo serviço está sob a mesma URI
  • Não pode ser armazenado em cache, pois o SOAP usa o método HTTP POST, o que é considerado não seguro, como vimos esse método pode alterar os dados de um recurso.

Conclusão

Conclusão

Apesar de antigo, o padrão SOAP é um dos mais utilizados ainda hoje, pois garante liberdade na hora da sua implementação, já que o mesmo não depende de um ambiente "fechado", ou seja, permite ser utilizado com qualquer linguagem de programação e vários protocolos, tais como HTTP e RPC.

Vale lembrar que a escolha entre SOAP ou outro padrão (ex.: REST) depende do cenário em questão, e não das vantagens de x ou y.
